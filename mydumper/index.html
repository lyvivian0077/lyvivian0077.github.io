    mydumper & myloader の使い方¶

    • mydumper主に2つのコマンド：
      • mydumperはバックアップ(dump)用。
      • myloaderは復旧(import)用。
    • mydumper Options
      • https://github.com/mydumper/mydumper#how-does-consistent-snapshot-work
      • https://github.com/mydumper/mydumper/blob/master/docs/mydumper_usage.rst

    単一のDBをDUMPする場合¶

    # 単一のDBをDUMPする場合
$ mydumper -u root -p root -h mysql_server  -B <対象DB名> -o <outputdir>

    複数のDBをDUMPする場合¶

    • 注意：

      • mysqldumpみたいなカンマ（,）で区切りして指定できないです。正規表現で複数のDBを指定する。

    • 全DBをdumpする場合 

      # zst圧縮率指定
/usr/bin/mydumper --events --routines --triggers --success-on-1146 --statement-size=8000000 --chunk-filesize=1024 --kill-long-queries --complete-insert --use-savepo ints --threads=8 --less-locking --trx-consistency-only --logfile /home/backup/mydumper.log  -v 3 --outputdir=/home/backup/20230925/ --defaults-file=/home/backup/mydumper.cnf --exec-per- thread '/usr/bin/zstd --ultra -22' --exec-per-thread-extension=".zst"

# zst圧縮率指定しない（デフォルト:3）
/usr/bin/mydumper --events --routines --triggers --success-on-1146 --statement-size=8000000 --chunk-filesize=1024 --kill-long-queries --complete-insert --use-savepoints --threads=8 --less-locking --trx-consistency-only --logfile /home/backup/mydumper.log  --compress=ZSTD -v 3 --outputdir=/home/backup/20230925/ --defaults-file=/home/backup/mydumper.cnf

# 圧縮なし
# mydumper -o /root/work/20220728_test

# 圧縮あり
# mydumper -o /root/work/20220728_test　-c

    • python用ソース抜粋

      def my_dumper(regex, cores, log_file, desc_path):
    """
    ログファイルは自動的に上書きされる
    exclude_dbs を除外して残りを全部dumpする
    regex: exclude dbs
    cores: threads
    DUMP_LOG_PATH + logfile: logfile
    desc_path: dump file dir
    return p.returncode, stdout_data, stderr_data
    (0, '', '')
    """
    command = ("nice -n "
               + str(NICE_LV) +
               " /usr/bin/mydumper"
               + regex +
               " --events --routines --triggers --success-on-1146"
               " --statement-size=8000000 --chunk-filesize=1024"
               " --kill-long-queries --complete-insert --use-savepoints"
               " --socket=/tmp/mysql.sock --defaults-file=/root/.my.cnf "
               " --threads="
               + cores
               + " --less-locking --trx-consistency-only --logfile "
               + DUMP_LOG_PATH + logfile
               + " --compress=ZSTD -v "
               + str(LOG_LV)
               + " --outputdir="
               + desc_path)
    return shell.runCmd(command)

    • Once can use --regex functionality, for example not to dump mysql and test databases: 

      mydumper --regex '^(?!(mysql\.|test\.))'

    • To dump only mysql and test databases 

      mydumper --regex '^(mysql\.|test\.)'

    • To not dump all databases starting with test: 
      mydumper --regex '^(?!(test))'
    • To dump specify tables in different databases (Note: The name of tables should end with $. related issue): 
      mydumper --regex '^(db1\.table1$|db2\.table2$)'
    • If you want to dump a couple of databases but discard some tables, you can do: 
      # Which will dump all the tables in db1 and db2 but it will exclude db1.table1 and db2.table2
mydumper --regex '^(?=(?:(db1\.|db2\.)))(?!(?:(db1\.table1$|db2\.table2$)))'

# 恢复备份文件中的全部 若表已存在则先删除
myloader -u root -p 123456 -o -d /backups/all3

# 从全备中恢复指定库
myloader -u root -p 123456 -s db1 -o -d /backups/all3

# 将某个数据库备份还原到另一个数据库中（目标库不存在则会新建）
myloader -u root -p 123456 -B recover_db1 -s db1 -o -d /backups/all3
myloader -u root -p 123456 -B recover_db1 -o -d /backups/db1

# 恢复时开启binlog（有备库的时候需要开启）
myloader -u root -p 123456 -e -o -d /backups/db1

# 无法直接还原单表 只能找到单表的sql文件 进入命令行source 执行
source db1.tb1-schema.sql 还原表结构
source db1.tb1.sql 还原表数据
    • mydumper备份过程 
      mydumper备份过程
1、连接目标数据库 / 対象DBに接続
2、通过show processlist来判断是否有长查询，根据参数long-query-guard和kill-long-queries决定退出或杀掉长查询 / show processlistにて長いクエリを確認し、引数long-query-guardとkill-long-queriesにてkillする。
3、主线程 / メインプロセス flush tables with read lock；start transaction with consistent snapshot
4、主线程读取当前时间点的二进制日志文件名和日志写入的位置并记录在metadata文件中，以供即时点恢复使用 / メインプロセスにて現在の時間のbinlogファイル名とbinlogの位置をmetadataファイルに記録する。
5、创建dump线程，缺省为4个　/　dumpスレットを作成、デフォルトは4
6、确定候选表，根据类别分别插入non_innodb_table、innodb_tables以及table_schemas链表　/　対象テーブルを確定し、種類別に対象リストに登録
7、将候选表通过g_async_queue_push加入任务队列(队列最后元素是thread shutdown)，由dump线程从队列中读取表信息并执行数据导出　/　対象をg_async_queue_pushにて転送リストに追加、dumpスレッドが対象リストを読み出し、dumpする。
8、备份完non_innodb_table后立即unlock tables解锁，以减少锁定时间　/　non_innodb_tableをバックアップ後、すぐにunlock tablesする。
9、等待dump InnoDB tables完成　/　InnoDBのdump完了待ち。

        mydumper X 早い 早い 早い 早い

      • メリット・デメリット

        メリット デメリット
        mydumper dumpしたものはsqlファイルです、復旧時DB毎にimportできる。バックアップ速度も早い。 圧縮なしのバックアップは早いですが、圧縮付きの場合が遅い、さらにdump中、対象テーブルがlockされたまま、操作できない。圧縮なしでバックアップし、その後圧縮(tar)すると、1サーバが半日以上かかります。
        xtrabackup 増量分だけバックアップするため、テーブル数少ない場合がとても早い。増量分のバックアップサイズの割と小さい。 復旧は全DBしか対応できないため、試験機にimport後、対象DBを抽出する方法しかない。手間かかる。対象サーバのDB/TBが多い場合、とても遅い上に、対象テーブルもlockされたまま操作できない。

    復旧方法主に2つあります。

    1. DBサーバ上の復旧作業 サービスDBサーバ上の直接復旧作業の場合は、サーバ上のバックアップデータから直接復旧する。 この方法はオススメしない、サービスサーバ上実行した場合、それなりの危険性もあるため、理解してから作業してください。（基本は方法２で実施する）

        还原整个实例
备份整个实例，包括触发器，事件，函数和存储过程


还原整个实例
myloader -u root -p root -S /tmp/mysql3307.sock -d /backup/mysql/bak/

mysql_upgrade 升级 slave 后还原备份
master

mydumper -u root -p root -S /tmp/mysql3306.sock -G -R -E -o /backup/mysql/bak
slave

先升级 mysql,否则会报 mysql.event, mysql.proc 这些表不存在的错误

mysql_upgrade -u root -p root -S /tmp/mysql3307.sock -P 3307
还原 3306 实例上备份的数据到 3307 实例上, 3307 实例是初始化的，没有数据写入的实例

myloader -u root -p root -S /tmp/mysql3307.sock -d /backup/mysql/bak
注意事项

slave
reset master;

还原单库
从备份的单个库文件中还原单库

myloader -u root -p root -h 192.168.2.249 -P 3307 -B zst -d /backup/mysql/bak/
从备份的真个实例文件中还原单库

myloader -u root -p root -h 192.168.2.249 -P 3307 -B zst -s zst -d /backup/mysql/bak/
还原多个库

备份多个库
mydumper -u root -p root -h 192.168.2.249 -P 3306 --regex 'zst|db3' -o /backup/mysql/bak

还原多个库
myloader -u root -p root -h 192.168.2.249 -P 3307 -d /backup/mysql/bak/
导入单库多表

备份单库多表
mydumper -u root -p root -h 192.168.2.249 -P 3306 -B zst -T sbtest1,sbtest2 -o /backup/mysql/bak

还原单库多表
myloader -u root -p root -h 192.168.2.249 -P 3307 -d /backup/mysql/bak/

当备份是实例级别的备份时，可以将需要的表的文件放到另一个目录中，还原时指定该目录即可

    2. お客さんからdumpファイル提供依頼の復旧作業

      • 直接サービスサーバ上のDB dump抽出はしない。
      • 対象バックアップを試験機に転送し、試験機上で作業する。 
        1. 対象のバックアップファイルを試験機に転送する。
[root@mysql57 backup]# scp -r mysqlxxx:/home/backup/mysqlxxx-dump_2022xxxx.tar ./

2. 対象バックアップファイルを解凍する。
[root@mysql57 backup]# tar xvf mysqlxxx-dump_2022xxxx.tar

3. 対象DBだけを試験機にimport
[root@mysql57 backup]# myloader --directory=2022xxxx  --threads=8 --verbose=3  -B 対象DB名 -s 対象DB名 -o -e

4. 対象DBをSQL形式のファイルとしてdumpする。
[root@mysql57 backup]# mysqldump --opt --databases  対象DB名 --hex-blob --quote-names --force --max_allowed_packet=128M --events --single-transaction --default-character-set=binary > 対象DB名.sql

5. dumpしたSQLファイルをｗｗｗサービスサーバに転送する。
[root@mysql57 ~]# scp 対象DB名.sql admin@server:/home/admin/

    mysqldumpの使い方¶

    • Export: 

      mysqldump -u root -p --all-databases > alldb.sql

    • Look up the documentation for mysqldump. You may want to use some of the options mentioned in comments: 

      mysqldump -u root -p --opt --all-databases > alldb.sql
mysqldump -u root -p --all-databases --skip-lock-tables > alldb.sql

    • 例: 

      # local
mysqldump \
--single-transaction \
--hex-blob \
--quote-names \
--events \
--force \
--max_allowed_packet=128M \
--default-character-set=binary \
--all-databases > alldatabases.dump

# remote
 mysqldump -h mysql_server -uroot -ppassword --single-transaction --hex-blob --quote-names --events --force --max_allowed_packet=128M --default-character-set=binary --all-databases > alldatabases.dump

    • Import: 

      mysql -u root -p < alldb.sql
